## 1. 배경 및 문제 정의

### 1.1 현재 상황

`schema_analyzer`가 LLM(`_analyze_db_structure`)을 사용하여 EAV 패턴을 자동 감지한다:

```
스키마 조회 → LLM 분석 → structure_meta 생성 → config/db_profiles/{db_id}.yaml 자동 저장
```

감지되는 EAV 패턴 정보:
```yaml
patterns:
  - type: eav
    entity_table: cmm_resource
    config_table: core_config_prop
    join_condition: "core_config_prop.configuration_id = cmm_resource.resource_conf_id"
    attribute_column: name           # ← 타입 구분 필드 (EAV의 Attribute)
    value_column: stringvalue_short  # ← 값 필드 (EAV의 Value)
```

### 1.2 핵심 문제: LLM이 EAV 구조를 정확히 파악하기 어려운 이유

EAV 테이블은 일반 정규화 테이블과 달리 **PK가 없거나 불분명한 경우가 많다**. 이로 인해 두 가지 핵심 정보를 LLM이 스키마만으로 정확히 감지하기 어렵다:

**문제 1: 테이블 간 JOIN 조건 감지 불확실**

EAV 구조에서는 entity 테이블과 EAV 테이블 간의 관계를 LLM이 자동 감지하기 매우 어렵다. 특히 **FK 제약이 아예 존재하지 않는 경우**(Polestar 등)에는 조인 조건을 스키마에서 유추하는 것이 사실상 불가능하다.

이때 테이블 간 관계를 파악하는 유일한 수단은 **값 기반 조인(value-based join)** — EAV 테이블의 값(value)이 entity 테이블의 특정 컬럼과 동일한 의미를 가지는 대응 관계를 아는 것이다.

```
Polestar 예시 (FK 없음, 값 기반 조인만 존재):

  core_config_prop 행: name='Hostname', stringvalue_short='server01'
  cmm_resource 행:     hostname='server01'

  → core_config_prop.name = 'Hostname'인 행의
    STRINGVALUE_SHORT 값이 cmm_resource.hostname과 동일한 값이다.
  → 즉: WHERE p.name = 'Hostname' AND p.stringvalue_short = r.hostname

이러한 "EAV 속성값 = entity 컬럼값" 대응 관계는
스키마만으로는 발견 불가능하며, 도메인 지식을 가진 운영자만 설정할 수 있다.
```

이 값 기반 조인 정보는 쿼리 생성 시 LLM에게 "어떤 EAV 속성이 어떤 entity 컬럼과 대응하는지" 힌트를 제공하여 올바른 SQL을 생성하도록 돕는다.

> **참고**: 일부 DB에서는 FK 제약이 존재할 수 있으며, 이 경우 `join_condition`(선택사항)으로 FK 조인을 별도 지정할 수 있다. 그러나 FK가 없는 DB에서는 `value_joins`가 테이블 간 관계를 정의하는 유일한 수단이다.

**문제 2: attribute_column (타입 구분 필드) 감지 불확실**

EAV 테이블에서 **어떤 컬럼이 "정보의 타입"을 정의하는지**(attribute_column)는 스키마 구조만으로 확실히 판별하기 어렵다.

| 상황 | 설명 |
|------|------|
| `name`이 타입 구분이 아닌 경우 | 어떤 테이블에서 `name`은 단순 식별자이고, `type` 또는 `category`가 진짜 구분 컬럼일 수 있음 |
| 다중 EAV 레벨 | 한 DB에 EAV 패턴이 여러 개 존재할 때, 각각의 attribute_column이 다를 수 있음 |
| 복합 구분 | attribute_column이 1개가 아닌 복합 키(name + category)일 수 있음 |
| 새 DB 온보딩 | LLM이 처음 보는 DB의 EAV 구조를 정확히 파악하기 어려움 |

**문제 1과 문제 2의 연결**: attribute_column을 알아야 값 기반 조인도 정의할 수 있다. Polestar의 경우:
- `core_config_prop.name`이 attribute_column (타입 구분)
- `name='Hostname'`일 때 `STRINGVALUE_SHORT` 값(`'server01'`)이 `cmm_resource.hostname`과 동일
- `name='IPaddress'`일 때 `STRINGVALUE_SHORT` 값이 `cmm_resource.ipaddress`와 동일
- 이 대응 관계를 모르면 올바른 EAV 피벗 쿼리 + 값 기반 조인 SQL을 생성할 수 없다

→ 따라서 프로필에는 attribute_column, known_attributes, **value_joins** 정보를 함께 정의해야 한다. 실제 Polestar 프로필(`config/db_profiles/polestar_pg.yaml`)을 이 계획과 함께 생성한다.

### 1.3 해결 방향: LLM 1차 분석 → 사용자 수동 보정

위 두 문제를 해결하기 위해 **LLM이 1차 분석한 결과를 초안으로 제공하고, 사용자(운영자)가 이를 기반으로 직접 profile을 보정**할 수 있도록 지원한다.

```
[새 DB 온보딩 워크플로우]

1. schema_analyzer가 LLM으로 구조 1차 분석 → config/db_profiles/{db_id}.yaml 자동 생성 (source: auto)
2. 운영자가 YAML을 검토:
   - attribute_column이 맞는지 확인/수정
   - value_joins 설정: EAV 속성값과 entity 컬럼 간 값 대응 관계 정의 (FK 없는 경우 필수)
   - join_condition 확인/수정 (FK가 있는 DB에서만, 선택사항)
   - known_attributes 목록 보강
3. source: manual 로 변경 → 이후 LLM 재분석 시에도 수동 설정 보호
```

이 방식은 LLM의 자동 감지 능력을 활용하면서도, 도메인 지식이 필요한 부분은 사람이 확정하는 하이브리드 접근이다.

### 1.4 현재 자동 생성 파일의 한계

`config/db_profiles/{db_id}.yaml`은 `# AUTO-GENERATED by structure analyzer` 헤더로 생성되며:
- **수동 편집 보호 없음**: LLM이 재분석하면 수동 편집 내용이 덮어씌워짐
- **머지 로직 없음**: 수동 설정과 자동 감지 결과를 합치는 로직이 없음
- **known_attributes 미포함**: 현재 YAML에는 EAV의 알려진 속성값 목록이 저장되지 않음

## 2. 접근 방식 비교

### 방안 A: 수동 설정 파일 + LLM 자동 감지 하이브리드 (권장)

기존 `config/db_profiles/{db_id}.yaml`을 확장하여 수동/자동 구분을 도입하고, 수동 설정이 자동 감지보다 항상 우선하도록 한다.

| 장점 | 단점 |
|------|------|
| 기존 인프라(YAML 파일, Redis 캐시) 100% 재사용 | YAML 파일 관리 필요 |
| LLM 감지 실패 시에도 안전하게 동작 | 새 DB마다 초기 설정 작성 필요 (하지만 LLM이 초안을 생성해줌) |
| 운영자가 도메인 지식을 명시적으로 인코딩 가능 | - |
| LLM이 초안을 자동 생성 → 운영자가 검증/보정만 하면 됨 | - |

### 방안 B: DB 스키마 어노테이션

DB 자체에 주석(COMMENT)으로 EAV 메타데이터를 기록하고, 스키마 조회 시 이를 읽어오는 방식.

| 장점 | 단점 |
|------|------|
| 데이터와 메타데이터가 같은 곳에 존재 | DB 쓰기 권한 필요 (D-003 위반 가능) |
| 별도 설정 파일 불필요 | 타사 DB(Polestar 등)의 COMMENT를 수정할 수 없는 경우 많음 |

### 방안 C: 샘플 데이터 카디널리티 기반 자동 감지

EAV 후보 테이블의 각 컬럼에 대해 카디널리티(고유값 수)를 분석하여, 낮은 카디널리티 컬럼을 attribute_column 후보로 제시.

| 장점 | 단점 |
|------|------|
| 완전 자동 | 카디널리티만으로는 판별 불충분 (status, flag 등도 낮은 카디널리티) |
| LLM 비용 절감 | 추가 DB 쿼리 필요 |
| - | 결국 사람의 확인이 필요하여 방안 A의 보조 수단 정도 |

### 결론

**방안 A(수동 설정 + LLM 보완)를 주 방식으로 채택하고, 방안 C(카디널리티 분석)를 LLM 감지의 보조 힌트로 활용한다.**

## 3. 설계

### 3.1 설정 파일 구조 확장

`config/db_profiles/{db_id}.yaml`의 포맷을 확장한다:

```yaml
# config/db_profiles/polestar_pg.yaml
# 수동 설정 항목은 LLM 자동 분석보다 항상 우선합니다.
# source: manual | auto | merged
source: manual

patterns:
  - type: eav
    entity_table: cmm_resource
    config_table: core_config_prop
    # join_condition: (선택사항) FK 제약이 있는 DB에서만 지정. Polestar는 FK가 없으므로 생략.
    attribute_column: name              # EAV의 Attribute (타입 구분 필드) — LLM 감지 어려움, 수동 확인 필수
    value_column: stringvalue_short     # EAV의 Value (값 필드)
    lob_value_column: stringvalue       # 선택사항: LOB 값 컬럼
    lob_flag_column: is_lob             # 선택사항: LOB 플래그 컬럼

    # 값 기반 조인 (value_joins): 테이블 간 관계를 정의하는 핵심 설정.
    # FK가 없는 DB에서는 이것이 entity-EAV 테이블 간 관계를 파악하는 유일한 수단이다.
    # 스키마만으로는 발견 불가능하며, 도메인 지식을 가진 운영자만 설정할 수 있다.
    value_joins:
      - eav_attribute: Hostname         # core_config_prop.name = 'Hostname'
        eav_value_column: stringvalue_short  # 이 속성의 값이 저장된 컬럼
        entity_column: hostname         # cmm_resource.hostname과 동일한 값
        description: "EAV Hostname 속성값 = cmm_resource.hostname"
      - eav_attribute: IPaddress
        eav_value_column: stringvalue_short
        entity_column: ipaddress
        description: "EAV IPaddress 속성값 = cmm_resource.ipaddress"

    known_attributes:                    # 이 EAV 테이블의 알려진 속성값 목록
      - name: OSType
        description: "운영체제 종류"
        synonyms: ["운영체제", "OS종류", "OS 타입"]
      - name: Vendor
        description: "제조사"
        synonyms: ["벤더", "제조사", "제조업체"]
      - name: Model
        description: "모델명"
        synonyms: ["모델", "장비모델"]
      - name: SerialNumber
        description: "시리얼 번호"
        synonyms: ["시리얼", "S/N", "일련번호"]
      - name: OSParameter
        description: "OS 파라미터"
        synonyms: ["OS파라미터", "OS 설정"]
      - name: AgentID
        description: "에이전트 ID"
        synonyms: ["에이전트ID", "에이전트"]
      - name: IPaddress
        description: "IP 주소"
        synonyms: ["IP주소", "IP"]
      - name: Hostname
        description: "호스트명 (EAV)"
        synonyms: ["EAV호스트명"]

  - type: hierarchy
    table: cmm_resource
    id_column: id
    parent_column: parent_resource_id
    type_column: resource_type
    name_column: hostname

query_guide: |
  Polestar DB는 cmm_resource(리소스 계층)와 core_config_prop(EAV 설정) 2개 테이블로 구성됩니다.
  서버 속성 조회 시 EAV 피벗 쿼리가 필요합니다.
  FK 제약이 없으므로, 테이블 간 관계는 값 기반 조인으로만 파악합니다.

  값 기반 조인:
    - core_config_prop.name='Hostname'인 행의 stringvalue_short = cmm_resource.hostname
    - core_config_prop.name='IPaddress'인 행의 stringvalue_short = cmm_resource.ipaddress
```

### 3.2 source 필드 의미

| source 값 | 의미 | 자동 덮어쓰기 |
|-----------|------|--------------|
| `manual` | 운영자가 직접 작성/편집 | **절대 덮어쓰기 않음** |
| `auto` | LLM 자동 생성 (현재 동작) | LLM 재분석 시 덮어쓰기 |
| `merged` | 수동 설정 기반 + LLM이 빈 부분을 채운 결과 | 수동 부분은 보존, 자동 부분만 갱신 |

### 3.3 머지 로직

`_save_structure_profile()` 수정:

```
1. 기존 YAML 파일 읽기
2. source == "manual" → 저장하지 않고 즉시 리턴 (수동 설정 보호)
3. source == "merged" → 수동 설정 항목 보존, LLM 결과로 빈 필드만 보충
4. source == "auto" 또는 파일 없음 → 현재 동작과 동일 (전체 덮어쓰기)
```

### 3.4 프로필 로드 시 우선순위

`schema_analyzer` 구조 분석 흐름 변경:

```
1. config/db_profiles/{db_id}.yaml 읽기
2. source == "manual" → LLM 분석 스킵, YAML 내용을 _structure_meta로 직접 사용
3. source == "merged" → YAML 내용 우선 사용, 빈 부분만 LLM 보충
4. source == "auto" 또는 파일 없음 → 기존 흐름 (LLM 분석 → 자동 저장)
5. 결과를 Redis 캐시에도 저장
```

### 3.5 known_attributes → Redis eav_name_synonyms 연동

현재 `eav_name_synonyms`는 Redis에서만 관리된다. 수동 프로필의 `known_attributes`에 정의된 속성과 synonyms를 Redis에 자동 로드하는 로직을 추가한다:

```
프로필 로드 시:
  for pattern in patterns where type == "eav":
    for attr in pattern.known_attributes:
      eav_name_synonyms[attr.name] = attr.synonyms
  → Redis synonyms:eav_names에 저장
```

이로써 field_mapper의 2.5단계 EAV synonym 매칭이 수동 설정된 known_attributes와 자동으로 연동된다.

### 3.6 value_joins → query_generator 프롬프트 연동

`value_joins`는 EAV 속성값과 entity 컬럼 간의 **값 대응 관계**를 정의한다. 이 정보는 query_generator의 프롬프트에 삽입하여 LLM이 올바른 WHERE/JOIN 조건을 생성하도록 돕는다.

**활용 시나리오**: "hostname이 server01인 서버의 OS정보를 조회해줘"

```
value_joins가 없을 때 (현재):
  LLM이 cmm_resource.hostname = 'server01'만 사용하거나,
  core_config_prop과의 관계를 잘못 추론

value_joins가 있을 때:
  프롬프트에 다음 힌트 삽입:
    "core_config_prop.name='Hostname'인 행의 stringvalue_short 값은
     cmm_resource.hostname과 동일한 값입니다."
  → LLM이 올바른 조인 조건과 필터를 생성
```

**프롬프트 삽입 방식**: `_build_structure_guide()` 함수에서 `value_joins`가 존재하면 query_guide에 값 대응 관계를 자동 추가한다.

```python
# value_joins를 query_guide 프롬프트에 변환
for vj in eav_pattern.get("value_joins", []):
    guide += (
        f"\n- {config_table}.{attr_col}='{vj['eav_attribute']}'인 행의 "
        f"{vj['eav_value_column']} 값은 "
        f"{entity_table}.{vj['entity_column']}과 동일한 값입니다."
    )
```

### 3.7 카디널리티 분석 보조 (선택사항, Phase 2)

LLM 자동 분석 시 보조 힌트로 카디널리티 정보를 제공:

```sql
-- EAV 후보 테이블의 각 VARCHAR 컬럼에 대해
SELECT '{column_name}' AS col,
       COUNT(DISTINCT {column_name}) AS cardinality,
       COUNT(*) AS total_rows
FROM {table_name}
LIMIT 1;
```

- `cardinality / total_rows` 비율이 낮은 컬럼 → attribute_column 후보
- 이 정보를 LLM 프롬프트에 삽입하여 감지 정확도 향상

## 8. 기대 효과

1. **테이블 간 관계의 확실한 설정**: FK가 없는 EAV 구조에서 `value_joins`로 값 기반 조인을 명시적으로 지정. EAV 속성값과 entity 컬럼의 대응 관계는 스키마에서 유추 불가능하므로 수동 설정이 필수. FK가 있는 DB에서는 `join_condition`도 선택적으로 지정 가능
2. **EAV attribute_column의 확실한 설정**: 도메인 지식을 가진 운영자가 타입 구분 필드를 명시적으로 지정
3. **새 DB 온보딩 간소화**: LLM이 1차 분석 초안을 자동 생성 → 운영자가 JOIN 조건/attribute_column/known_attributes를 검증/보정 → `source: manual`로 확정
4. **known_attributes 일원화**: YAML 파일 하나로 EAV 속성, synonyms, description을 모두 관리
5. **LLM 비용 절감**: manual 프로필이 있으면 구조 분석 LLM 호출 자체를 스킵
6. **안정성 향상**: LLM의 FK/JOIN 감지 오류 및 attribute_column 감지 오류로 인한 잘못된 SQL 생성 위험 제거
