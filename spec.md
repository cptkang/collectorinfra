# 인프라 데이터 조회 에이전트 요건 정의서

## 1. 프로젝트 개요

### 1.1 목적

사용자가 자연어로 인프라 정보를 질의하면, LLM이 DB 스키마를 분석하여 SQL을 자동 생성·실행하고, 결과를 정리하여 자연어 응답 또는 문서(Excel/Word) 형태로 제공하는 에이전트를 구축한다.

### 1.2 핵심 워크플로우

```
사용자 자연어 질의 또는 양식 파일 업로드
    ↓
LLM이 DB 스키마 분석 (DBHub 통해 메타데이터 조회)
    ↓
SQL 쿼리 자동 생성 및 검증
    ↓
DB 실행 및 데이터 수집
    ↓
LLM이 결과 정리 → 자연어 응답 생성 또는 양식 파일 작성
```

### 1.3 기술 스택

| 구분 | 기술 | 비고 |
|------|------|------|
| 에이전트 프레임워크 | **LangGraph** | 노드/엣지 기반 상태 머신 |
| LLM | Claude / GPT 등 | 프롬프트 기반 SQL 생성 및 응답 정리 |
| DB 연결 | **DBHub** (MCP 서버) | 제로 의존성, 다중 DB 지원 |
| 문서 처리 | python-docx, openpyxl | Word/Excel 양식 파싱 및 생성 |
| API 서버 | FastAPI (선택) | 사용자 인터페이스 제공 |

### 2.2 LangGraph 노드 정의

| 노드 | 역할 | 입력 | 출력 |
|------|------|------|------|
| **input_parser** | 사용자 입력(자연어/파일) 분석 및 의도 파악 | 사용자 메시지, 업로드 파일 | 파싱된 요구사항(질의 항목, 양식 구조) |
| **schema_analyzer** | DBHub를 통해 DB 스키마 조회 및 관련 테이블 식별 | 파싱된 요구사항 | 관련 테이블/컬럼 목록, 샘플 데이터 |
| **query_generator** | LLM을 이용한 SQL 쿼리 생성 | 요구사항 + 스키마 정보 | SQL 쿼리 문자열 |
| **query_validator** | 생성된 SQL의 문법·안전성 검증 | SQL 쿼리 | 검증 결과 (통과/재생성 필요) |
| **query_executor** | DBHub를 통해 SQL 실행 | 검증된 SQL 쿼리 | 쿼리 결과 데이터 |
| **result_organizer** | 쿼리 결과를 사용자 요구에 맞게 정리 | 쿼리 결과 + 원본 요구사항 | 정리된 데이터 |
| **output_generator** | 자연어 응답 생성 또는 양식 파일(Excel/Word) 작성 | 정리된 데이터 + 양식 템플릿 | 최종 응답 또는 파일 |

### 2.3 엣지 (제어 흐름)

```
START → input_parser → schema_analyzer → query_generator → query_validator
    → [조건부 엣지]
        ├─ 검증 통과 → query_executor → result_organizer → output_generator → END
        └─ 검증 실패 → query_generator (재시도, 최대 3회)
```

추가 조건부 엣지:

- `query_executor`에서 에러 발생 시 → `query_generator`로 회귀 (에러 메시지 포함하여 재생성)
- `result_organizer`에서 데이터 부족 판단 시 → `query_generator`로 회귀 (추가 쿼리 생성)

### 2.4 State 스키마

```python
from typing import TypedDict, Optional
from langgraph.graph import MessagesState

class AgentState(TypedDict):
    # 사용자 입력
    user_query: str                     # 자연어 질의
    uploaded_file: Optional[bytes]      # 업로드된 양식 파일
    file_type: Optional[str]           # "xlsx" | "docx" | None

    # 파싱 결과
    parsed_requirements: dict           # 추출된 요구사항
    template_structure: Optional[dict]  # 양식 구조 정보

    # DB 관련
    relevant_tables: list[str]          # 관련 테이블 목록
    schema_info: dict                   # 스키마 상세 정보
    generated_sql: str                  # 생성된 SQL
    query_results: list[dict]           # 쿼리 실행 결과

    # 제어
    retry_count: int                    # 재시도 횟수
    error_message: Optional[str]        # 에러 메시지

    # 출력
    final_response: str                 # 자연어 응답
    output_file: Optional[bytes]        # 생성된 파일
```

## 3. DBHub 연동 명세

### 3.1 DBHub 설정 (TOML 구성)

```toml
[[sources]]
name = "infra_db"
type = "postgresql"          # 또는 mysql, mariadb 등
connection = "postgresql://user:password@host:5432/infra_db"
readonly = true              # 안전을 위해 읽기 전용 권한
query_timeout = 30           # 쿼리 타임아웃 (초)

[[tools]]
name = "execute_sql"
sources = ["infra_db"]
readonly = true
max_rows = 10000             # 최대 반환 행 수
```

### 3.2 DBHub 활용 도구

| 도구 | 용도 | 사용 노드 |
|------|------|----------|
| `search_objects` | 테이블 목록 조회, 스키마 탐색 | schema_analyzer |
| `execute_sql` | SQL 쿼리 실행 | query_executor |
| Custom Tool (선택) | 자주 사용하는 정형 쿼리 사전 정의 | query_executor |

### 3.3 보안 고려사항

- DB 연결은 **읽기 전용(readonly)** 모드로 제한
- 쿼리 타임아웃 설정으로 장시간 쿼리 방지
- max_rows 제한으로 과도한 데이터 반환 차단
- SSH 터널링을 통한 안전한 원격 DB 접속 지원
- 사용자 입력으로부터의 SQL 인젝션 방지 (LLM 생성 SQL도 검증 필수)

## 4. 인프라 데이터 모델

### 4.1 주요 데이터 도메인

에이전트가 조회할 인프라 데이터의 범위는 다음과 같다.

| 도메인 | 설명 | 주요 지표 |
|--------|------|----------|
| **서버 정보** | 물리/가상 서버 기본 정보 | 호스트명, IP, OS, 위치, 용도 |
| **CPU (Core)** | CPU 자원 현황 | 코어 수, 사용률, 할당량 |
| **메모리 (Memory)** | 메모리 자원 현황 | 총 용량, 사용량, 사용률 |
| **디스크 (Disk)** | 스토리지 현황 | 총 용량, 사용량, 파티션별 현황 |
| **네트워크 (NW Traffic)** | 네트워크 트래픽 현황 | 인바운드/아웃바운드 트래픽, 대역폭 |

### 4.2 예상 테이블 구조 (참고)

아래는 예상되는 DB 테이블 구조이며, 실제 스키마는 DBHub의 `search_objects`를 통해 런타임에 동적으로 파악한다.

```
servers             - 서버 기본 정보 (id, hostname, ip, os, location, purpose ...)
cpu_metrics         - CPU 지표 (server_id, timestamp, core_count, usage_pct ...)
memory_metrics      - 메모리 지표 (server_id, timestamp, total_gb, used_gb ...)
disk_metrics        - 디스크 지표 (server_id, timestamp, mount_point, total_gb, used_gb ...)
network_metrics     - 네트워크 지표 (server_id, timestamp, interface, in_bytes, out_bytes ...)
```

## 5. 양식 기반 문서 생성 기능

### 5.1 지원 파일 형식

| 형식 | 입력(양식 파싱) | 출력(결과 작성) |
|------|----------------|----------------|
| Excel (.xlsx) | O | O |
| Word (.docx) | O | O |

### 5.2 양식 처리 워크플로우

```
1. 사용자가 양식 파일(Excel/Word) 업로드
    ↓
2. input_parser 노드에서 양식 구조 분석
   - Excel: 시트별 헤더, 셀 위치, 데이터 영역 파악
   - Word: 표(Table) 구조, 플레이스홀더({{변수명}}) 파악
    ↓
3. LLM이 양식 항목과 DB 스키마를 매핑
   - 양식 헤더/플레이스홀더 → DB 컬럼 매핑
    ↓
4. 매핑 기반으로 SQL 쿼리 생성 및 실행
    ↓
5. 쿼리 결과를 양식 구조에 맞게 채워넣기
    ↓
6. 완성된 파일을 사용자에게 전달
```

### 5.3 양식 매핑 규칙

- **Excel 양식**: 첫 번째 행(또는 지정 행)을 헤더로 인식하고, 데이터 행 영역에 조회 결과를 채움
- **Word 양식**: `{{placeholder}}` 패턴 또는 표(Table)의 빈 셀을 인식하여 데이터를 삽입
- LLM이 양식 항목명과 DB 컬럼명 간의 의미적 매핑을 수행 (예: "서버명" → `servers.hostname`)

### 5.4 양식 파싱 상세

#### Excel 파싱 전략

```python
# 주요 파싱 로직 개요
- 시트 목록 및 각 시트의 구조 파악
- 헤더 행 자동 탐지 (첫 번째 비어있지 않은 행)
- 데이터 영역 범위 식별
- 병합 셀, 서식 정보 보존
- 수식이 포함된 셀은 유지하고 데이터 셀만 채움
```

#### Word 파싱 전략

```python
# 주요 파싱 로직 개요
- 문서 내 표(Table) 구조 탐지
- {{placeholder}} 패턴의 변수 추출
- 표 헤더 행과 데이터 행 구분
- 문서 스타일 및 서식 보존
```

## 6. LangGraph 에이전트 상세 설계

### 6.1 노드별 프롬프트 전략

#### input_parser 프롬프트

```
역할: 사용자의 입력(자연어 질의 또는 양식 파일)을 분석하여 구조화된 요구사항을 추출한다.

입력 분석 항목:
- 조회 대상: 어떤 인프라 데이터를 원하는가 (서버, CPU, 메모리, 디스크, 네트워크)
- 필터 조건: 특정 서버, 기간, 임계값 등
- 출력 형식: 자연어 응답 / Excel / Word
- 양식 파일이 있는 경우: 양식 구조 및 필요 데이터 항목
```

#### query_generator 프롬프트

```
역할: 주어진 스키마 정보와 사용자 요구사항을 기반으로 정확한 SQL 쿼리를 생성한다.

규칙:
- SELECT 문만 생성 (INSERT, UPDATE, DELETE 금지)
- 테이블 및 컬럼명은 반드시 스키마 정보에 존재하는 것만 사용
- 대량 데이터 조회 시 LIMIT 절 포함
- 필요 시 JOIN, GROUP BY, 집계 함수 활용
- 쿼리 설명 주석 포함
```

#### query_validator 검증 항목

```
1. SQL 문법 검증
2. SELECT 문 여부 확인 (DML/DDL 차단)
3. 참조 테이블/컬럼 존재 여부 확인
4. LIMIT 절 존재 여부 (대량 조회 방지)
5. 잠재적 성능 이슈 확인 (전체 테이블 스캔 등)
```

### 6.2 에러 핸들링 전략

| 에러 유형 | 처리 방식 |
|----------|----------|
| SQL 문법 오류 | 에러 메시지 포함하여 query_generator로 재시도 (최대 3회) |
| DB 연결 실패 | 사용자에게 연결 실패 알림, 재시도 안내 |
| 타임아웃 | 쿼리 최적화 후 재시도, 불가 시 사용자 알림 |
| 결과 없음 | 조건 완화 제안 또는 "해당 데이터 없음" 응답 |
| 양식 파싱 실패 | 지원하지 않는 양식 구조임을 안내 |

### 6.3 체크포인트 및 대화 관리

- LangGraph의 체크포인트 기능을 활용하여 대화 상태 유지
- 멀티턴 대화 지원: 이전 조회 결과를 참조한 후속 질의 가능
- 스레드 기반 세션 분리: 다중 사용자 동시 지원

| F-18 | 운영자 로그인 | 운영자 전용 화면에 ID/PW 기반 인증으로 접근할 수 있다 (사용자 화면과 분리) |
| F-19 | 환경변수 설정 UI | 운영자가 .env 파일의 설정값을 Web UI를 통해 조회/수정할 수 있다 |
| F-20 | DB 연결 설정 UI | 운영자가 DB connection 정보(호스트, 포트, DB명, 사용자, 비밀번호 등)를 Web UI에서 입력/수정할 수 있다 |

## 13. UI 화면 설계

### 13.1 사용자 화면

사용자 화면은 인증 없이 접근 가능한 메인 인터페이스이다.

#### 13.1.1 화면 구성

- **프롬프트 입력 영역**: 자연어 질의를 입력하는 텍스트 입력 필드 (여러 줄 입력 가능)
- **파일 첨부 영역**: Excel(.xlsx) 또는 Word(.docx) 양식 파일을 드래그앤드롭 또는 파일 선택으로 업로드
- **실행 버튼**: 프롬프트와 첨부 파일을 서버에 전송하여 처리 시작
- **결과 표시 영역**: LLM 응답(자연어 텍스트)을 표시하고, 생성된 파일이 있으면 다운로드 링크 제공

#### 13.1.2 동작 흐름

```
1. 사용자가 프롬프트를 입력한다 (필수)
2. 필요 시 양식 파일을 첨부한다 (선택)
3. "실행" 버튼을 클릭한다
4. 서버에서 처리 중임을 로딩 표시로 안내한다
5. 처리 완료 시 결과를 텍스트로 표시하고, 생성 파일은 다운로드 가능하게 제공한다
```

### 13.2 운영자 화면

운영자 화면은 ID/PW 인증을 통해 접근하는 관리용 인터페이스이다.

#### 13.2.1 인증

- 운영자 계정은 환경변수(ADMIN_USERNAME, ADMIN_PASSWORD)로 설정한다
- 로그인 폼에서 ID/PW를 입력하여 인증한다
- 인증 성공 시 JWT 토큰 또는 세션 기반으로 상태를 유지한다

#### 13.2.2 환경변수 설정 화면

- .env 파일에 정의된 설정값을 목록으로 표시한다
- 각 설정값을 인라인 편집하거나 수정 폼을 통해 변경할 수 있다
- 변경 사항은 .env 파일에 저장되고 앱 재시작 없이 가능한 경우 즉시 반영한다
- 민감 설정값(API 키, 비밀번호 등)은 마스킹 표시하며 수정 시에만 입력 가능하다

#### 13.2.3 DB 연결 설정 화면

- DB 연결에 필요한 정보를 입력/수정할 수 있는 폼을 제공한다:
  - DB 유형 (PostgreSQL, MySQL, MariaDB 등)
  - 호스트
  - 포트
  - 데이터베이스명
  - 사용자명
  - 비밀번호
- "연결 테스트" 버튼으로 입력한 정보의 유효성을 확인할 수 있다
- 저장 시 .env 및 dbhub.toml 설정 파일을 업데이트한다

## 8. 비기능 요건

### 8.1 성능

- 단순 조회 응답 시간: 10초 이내
- 복합 조회(JOIN, 집계) 응답 시간: 30초 이내
- 양식 파일 생성: 60초 이내
- 동시 사용자 지원: 최소 10명

### 8.2 보안

- DB 접근은 읽기 전용으로 제한
- 생성된 SQL에 대한 검증 레이어 필수
- 사용자 인증/인가 체계 적용
- 민감 데이터(비밀번호, 접근키 등) 마스킹 처리
- 감사 로그: 모든 쿼리 실행 이력 기록

### 8.3 안정성

- LLM API 호출 실패 시 재시도 로직 (exponential backoff)
- DB 연결 풀링 및 헬스체크
- 체크포인트를 통한 상태 복구 가능

### 8.4 확장성

- 새로운 DB 소스 추가가 DBHub TOML 설정만으로 가능
- 새로운 양식 형식(PDF 등) 추가 시 output_generator 노드만 확장
- LLM 모델 교체 시 프롬프트 레이어만 수정

## 9. 개발 단계별 계획

### Phase 1: 기본 자연어 → SQL 조회 파이프라인

- Human-in-the-loop (쿼리 승인) 기능
- 양식 템플릿 관리 기능
- 감사 로그 및 모니터링

## 11. 인프라 질의 예시

### 자연어 질의 예시

| 질의 | 예상 동작 |
|------|----------|
| "전체 서버의 CPU 사용률 현황을 알려줘" | servers + cpu_metrics 조인하여 서버별 최근 CPU 사용률 조회 |
| "메모리 사용률이 80% 이상인 서버 목록" | memory_metrics에서 사용률 80% 이상 필터링 |
| "지난 일주일간 네트워크 트래픽 Top 10 서버" | network_metrics에서 최근 7일 트래픽 집계 후 상위 10개 |
| "디스크 사용률 90% 이상인 서버의 상세 정보" | disk_metrics + servers 조인, 90% 이상 필터링 |
| "서버 A의 최근 한 달 리소스 사용 추이" | 서버명으로 필터 후 CPU/메모리/디스크/NW 시계열 데이터 조회 |

### 양식 기반 출력 예시

사용자가 아래와 같은 Excel 양식을 업로드하면:

```
| 서버명 | IP | CPU 코어 | CPU 사용률 | 메모리(GB) | 메모리 사용률 | 디스크(GB) | 디스크 사용률 |
|--------|-----|---------|-----------|-----------|-------------|-----------|-------------|
|        |     |         |           |           |             |           |             |
```

에이전트가 DB에서 데이터를 조회하여 양식을 채운 Excel 파일을 반환한다.
